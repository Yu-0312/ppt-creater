# 視覺生產與品質細則（Step 2–5 取用）

> 本檔的版面工藝原則，參考並改寫自 guizang-ppt-skill（作者 歸藏，https://github.com/op7418/guizang-ppt-skill）多輪實戰迭代的心得，整理成與本工作流通用的版本。
> 本檔只保留**跨工具通用的判斷**；該 skill 那套成套設計系統（電子雜誌風／瑞士國際主義風的模板、主題色、版型骨架、元件庫、驗版腳本）仍由它維護。若它已安裝，生成階段可直接交棒給它（見第 10 節）。

## 1. 生成前預檢（最能防崩版）

寫任何一頁之前，先列一張規劃表，每頁一列：`頁碼 → 版型 → 為什麼選它 → 圖片槽位`。

然後確認：你要用的每個版型／元件，背後的樣式（CSS class）都已存在於模板裡。**最常見的崩版原因，就是用了模板沒定義的 class**，瀏覽器 fallback 成預設樣式 → 標題字體錯、卡片擠成一團、圖片掉到頁尾。預檢一次，省下逐頁重修。

## 2. 視覺節奏（避免看到睡著）

- 每頁標記基調：`封面/幕封(hero)`、`深色`、`淺色`。
- 連續 3 頁以上同基調 = 視覺疲勞，不允許。
- 8 頁以上至少各有 1 個深色 hero 與 1 個淺色 hero。
- 每 3–4 頁插入一個 hero 頁（封面／章節／大引言／問題頁）喘口氣。
- 封面與結語深色滿版、前後呼應。

生成後快速掃一遍各頁基調，人工確認節奏再交付。

## 3. 中文大標題字號分檔（投屏必做）

中文方塊字視覺面積大，不能直接套英文 hero 字號。先分檔：

| 標題形態 | 建議字號 |
|---|---|
| 1 行，≤8 字 | `min(6.4vw, 11.2vh)` |
| 2 行，每行 ≤8 字 | `min(5.8vw, 10.2vh)` |
| 2 行，任一行 9–12 字 | `min(5.2vw, 9.2vh)` |
| 3 行或更長 | 先改寫標題；不得已 `min(4.6vw, 8.2vh)` |

標題擠到圖片或正文時，先壓縮文案，再降字號，不要把下面內容推到頁尾硬塞。
大字務必**雙約束限高**：`font-size: min(Xvw, Yvh)`——只用 vw 在標準 16:9 螢幕會溢出。

## 4. 投屏最小字號與字重階梯

小字不能按網頁註釋的 10–12px 寫。下限：

| 文字類型 | 最小字號 |
|---|---|
| 正文／主要說明 | 18px |
| 卡片描述／列表／時間線說明／圖注 | 16px |
| meta／kicker／mono label／圖表標籤 | 14px |

放不下就刪文案、拆兩頁、換版型，不要壓字級。

字重階梯「越大越細、越小越粗」：超大字（≥8vw）用 200 ExtraLight；章節標題 200–300；中型標題 300–400；正文 400–500；小字（13–15px）500–600。**同頁內，字越小字重必須 ≥ 字越大者**；16px 小字最低 400、別用 300。

## 5. 圖片工藝

- 只用標準比例：21:9 / 16:9 / 16:10 / 4:3 / 3:2 / 1:1；不要複製原圖奇葩比例（如 2592/1798）。
- 圖片網格用**固定高度**（如 `height:26vh`），不要用 `aspect-ratio`（會撐破版面）。
- 圖片／caption／時間線標籤的最低處，不要碰到底部翻頁區；需要貼底時用安全區類（如 `.nav-safe-bottom`），不要手寫 `bottom:2vh`。
- 圖片預設貼頂（grid + `align-items:start`），不要 `align-self:end` 讓它滑到頁尾。
- 命名：`images/` 下用 `{頁號}-{語義}.{ext}`，頁號補零、語義用英文（`01-cover.jpg`／`05-dashboard.png`）；替換用同名覆蓋最穩。

## 6. 鎖定調色盤（風格規格的另一個選項）

本工作流 Step 2 預設「給 4 個色碼」。若想要更穩的美學，可改用**鎖定預設盤**策略：準備幾套已調好的配色（背景／主文字／強調／點綴），只讓使用者**選一套、不接受任意 hex**——配色搭錯畫面瞬間變醜，鎖盤保護美學比給自由更重要。一份 deck 只用一套，不中途換色、不混搭。

## 7. 渲染後逐頁目視核對（不只看代碼）

代碼只證明 class 與結構存在，不證明版面舒服。生成後務必打開網頁逐頁看：

1. 等入場動效穩定（1–2 秒）再判斷，別把動畫中間態當版面問題。
2. 先看視覺：大標題字重、標題與內容間距、圖片是否與正文對齊、最低處是否碰到翻頁組件。
3. 再看代碼：版型與內容形狀是否匹配（沒把數據專用版型拿來講概念，也沒把可選組件堆成裝飾）。
4. 別扭時先判斷是「版型選錯／必選組件缺／可選組件濫用／間距安全區問題」，不要直接加 margin 硬救。

## 8. P0 自檢清單（交付前必過）

- [ ] 沒有用到模板未定義的 class（已預檢）
- [ ] 視覺節奏：沒有連續 3 頁同基調、hero 頁有穿插
- [ ] 中文大標題有分檔、雙約束限高、沒有 1 字 1 行
- [ ] 投屏字級達下限（正文 ≥18px、註腳 ≥14px）
- [ ] 圖片用標準比例、固定高度、沒碰到頁尾翻頁區
- [ ] 強調色每頁 ≤2 處、整份只一個主強調色系
- [ ] 標題襯線／內文無襯線分工正確、用圖標而非 emoji
- [ ] 圖表數字都經過 Step 4 重算、不一致已標註待核
- [ ] 已打開網頁逐頁目視核對過

## 9. 沒有大綱時：敘事弧模板

```
鉤子(Hook)     1 頁    ：拋一個反差／問題／硬數據讓人停下來
定調(Context)  1–2 頁  ：背景／你是誰／為什麼講這個
主體(Core)     3–5 頁  ：核心內容
轉折(Shift)    1 頁    ：打破預期／提出新觀點
收束(Takeaway) 1–2 頁  ：金句／懸念／行動建議
```
頁數參考：15 分鐘 ≈ 10 頁、30 分鐘 ≈ 20 頁、45 分鐘 ≈ 25–30 頁。

## 10. 成套設計系統（可選接力，強烈推薦）

生成（Step 3）這一步，若有專做視覺生產的 deck skill，**直接交棒給它**，不要從零手寫——本工作流守住 Step 1–2（內容清點＋敏感攔截＋風格規格）與 Step 4–5（圖表數據驗算＋驗收），也就是「該想清楚什麼、數據能不能信」。常見的兩個夥伴，定位不同：

- **guizang-ppt-skill ── 深度型（2 種精裝風格）**：電子雜誌風／瑞士國際主義風，附完整模板、鎖定主題色、10+22 種登記版型、元件庫、Motion 動效與驗版腳本（瑞士風交付前跑 `scripts/validate-swiss-deck.mjs`）。要那兩種風格、要最穩最精緻時用它。
- **frontend-slides ── 廣度型（30+ 種風格＋端到端產線）**：12 套預設＋34 個 bold 模板（含 mood/tone/formality/density/scheme 選風格 metadata）、固定 16:9 舞台、PPTX 匯入、瀏覽器內編輯、一鍵部署 URL／匯出 PDF。要多樣風格、要「先給看再選」、要分享連結或 PDF 時用它。

選用建議：**要那兩個精裝風格 → guizang；要其他多樣風格或完整分享流程 → frontend-slides；只是快速出一版 → 本工作流自己生成。** 三者都不重造對方維護中的資產。

## 11. 「先給看再選」風格探索（給說不出喜好的人）

多數人講不出設計偏好，但看到就知道喜不喜歡。所以 Step 2 除了「給色碼」，更好的作法是：

- 依用途、受眾、氣氛、密度，**直接生成 2–3 張真實的單頁預覽**（不同字體、配色、動效、版面），打開讓使用者挑。
- 預覽要像「這份簡報真正的第一頁」，**不要**在投影片上印出 `預覽`／`風格 A`／`option`／檔名／需求註記這類流程文字——那些只放在對話裡。
- 建議組合：1 個保守安全款＋1 個大膽特色款＋1 個外卡（針對這個場合自由設計）。三款要明顯不同，使用者才能用視覺反應而不是用詞彙選擇。

挑完再進 Step 3 生成。這比一開始就逼使用者講清楚配色快得多，也更不容易做出他不愛的東西。

**現成風格庫（本 skill 已內建）**：`bold-template-pack/` 收錄 34 種風格的設計規格。挑風格時的省力流程（漸進式載入，別一次全讀）：

1. 先讀 `bold-template-pack/selection-index.json`，依 `mood`／`tone`／`formality`／`density`／`scheme`／`best_for`／`avoid_for` 篩 1–2 個候選。
2. 只讀候選的 `templates/<slug>/preview.md` 做標題頁預覽。
3. 使用者選定後，才讀那一個 `templates/<slug>/design.md` 當設計配方（字體、配色、版面語彙、間距節奏、元件文法）。
4. 把 `design.md` 當「配方」而非「內容」——保留它的設計語彙，但投影片內容仍以使用者的素材為主，別照抄它的示範文字。

> 出處與授權：`bold-template-pack/` 來自 frontend-slides（作者 Zara Zhang，MIT），上游為 `zarazhangrui/beautiful-html-templates`。散布本 skill 時務必保留 `bold-template-pack/LICENSE` 與根目錄 `CREDITS.md`；上 GitHub 時於 README 增設 Credits 段落。詳見 `CREDITS.md`。

## 12. 內容密度模式（動手前先問一句）

先問：這是**演講型**還是**閱讀型**？兩者設計完全不同：

| 模式 | 適合 | 設計行為 |
|---|---|---|
| 低密度／演講型 | 公開演講、發表會、現場講解 | 一頁一個重點、大字、強層級、留白多、每頁 1–3 點，頁數可以多 |
| 高密度／閱讀型 | 報告、講義、非同步審閱、內部文件 | 每頁更自足、結構化網格／表格／註解、4–8 點或 4–6 卡、間距較緊但仍刻意 |

底線不變：不捲動、不溢出、不重疊、字不小於可讀下限。內容超過該模式就**拆成更多頁**，不要硬壓到擠。

## 13. 固定 16:9 舞台（防破版的技術底）

最穩的做法：每頁都在一個固定的 **1920×1080 舞台**裡編排，整個舞台等比縮放去貼合視窗；手機上寧可上下／左右留黑邊（letterbox），**也不要重排內容**。

- 不要用響應式斷點把投影片內容在手機上重新排版。
- 用 1920×1080 設計尺寸的固定內部尺寸；切換頁面用 visibility／opacity／pointer-events，不要用 `display:none/block`（後面的 layout class 可能蓋掉、讓所有頁同時冒出來）。
- 加上 `prefers-reduced-motion` 支援。
- 別直接對 CSS 函式取負（`-clamp()` 會被忽略），用 `calc(-1 * clamp(...))`。

## 14. 避免「AI slop」設計原則

預設輸出容易收斂成一眼就知道是 AI 生的通用感。刻意避開：

- **字體**：別用 Inter／Roboto／Arial／系統字；挑有個性、能撐起美感的字（投影片要襯線標題＋無襯線內文的分工）。也別每次都收斂到同一個「安全字」（如 Space Grotesk）。
- **配色**：承諾一套有主色＋銳利強調的調性，別用「白底紫漸層」這種陳腔濫調，也別平均分配多色。可從 IDE 主題、文化美學取靈感。
- **動效**：一個編排良好的入場（staggered reveal、animation-delay）比一堆零散微互動更有感；聚焦高衝擊時刻。
- **背景**：用漸層、幾何、紋理營造氛圍與層次，不要永遠純色底。
- 整體要像「為這個場合量身設計」，不是套版。

## 15. 截圖驗收（比 scrollHeight 可靠）

光檢查 `scrollHeight` 不夠——網格面板可能視覺上互相覆蓋卻沒觸發捲動。交付前用瀏覽器**實際截圖**逐頁看：在 1280×720 與一個手機視窗各截一次，確認①舞台維持 16:9、②文字沒溢出卡片、③面板沒重疊、④最低處沒碰到翻頁組件。改完任何一頁都要重截重驗。

## 16. 可選的輸入與輸出

- **輸入｜PPTX 匯入**：若來源是現成 .pptx，可先抽出文字／圖片／講者備註，再走本工作流（frontend-slides 有 `scripts/extract-pptx.py` 可用）。
- **輸出｜分享與匯出**：成品可部署成手機也能開的線上連結，或匯出 PDF 供 email／列印（frontend-slides 有 `scripts/deploy.sh`／`scripts/export-pdf.sh`）。PDF 是靜態快照、不含動畫，先告知使用者。
- **瀏覽器內編輯**：產出可保留「滑到左上角或按 E 進編輯、點文字直接改、Ctrl+S 存」的即時編輯，方便上台前微調。
